Especificação Funcional Completa — Banking API Training System

1. Introdução

1.1 Propósito

Este documento descreve de forma detalhada os requisitos funcionais, não funcionais, regras de negócio e fluxos principais do Banking API Training System, um sistema didático utilizado para demonstrar o ciclo de vida de desenvolvimento de software (SDLC) e práticas de QA.

1.2 Público-alvo

  • Desenvolvedores backend e frontend.
  • Analistas de QA (júnior a pleno).
  • Instrutores/mentores que utilizarão o sistema em treinamentos.

1.3 Escopo

  • API REST (Banking API).
  • Frontends web (desktop e mobile-like).
  • Collection de testes de API (Postman).
  • Artefatos de QA (casos de teste, plano de teste, etc., documentados em outro local).

2. Visão Geral de Arquitetura

  • Backend
    • Node.js + Express.
    • Memória em processo (sem banco externo).
    • Endpoints sob o path /accounts e sub-recursos.
  • Frontend Web
    • Aplicação estática em frontend-web/index.html.
    • Consome a API em http://localhost:3000.
  • Frontend Mobile-like
    • Aplicação estática em frontend-mobile/index.html.
    • Layout simulando aplicação mobile (390px de largura).
  • Documentação
    • Swagger UI em /api-docs.

3. Requisitos Funcionais da API

3.1 Modelo de Dados — Conta

Nome da entidade: Account

Campo Tipo Obrigatório Descrição
idstringSimIdentificador único (UUID)
namestringSimNome do titular da conta
emailstringSimEmail do titular
cpfstringSimCPF único do titular
balancenumberSimSaldo atual da conta
createdAtstringSimData/hora de criação em formato ISO

Regras gerais:

  • cpf deve ser único.
  • balance nunca pode ser negativo.
  • createdAt é definido automaticamente pelo sistema.

3.2 Endpoint: Criar Conta

  • Método: POST
  • URL: /accounts

Body (JSON):

{
  "name": "João Silva",
  "email": "joao@email.com",
  "cpf": "111.111.111-11",
  "initialBalance": 500
}

Validações:

  • name, email, cpf, initialBalance obrigatórios.
  • initialBalance >= 0.
  • cpf não pode existir em outra conta.

Respostas:

  • 201 Created: conta criada com balance = initialBalance.
  • 400 Bad Request: campos ausentes, initialBalance inválido ou CPF duplicado.

3.3 Endpoint: Listar Contas

  • Método: GET
  • URL: /accounts
  • Comportamento: retorna lista de todas as contas.
  • Resposta: 200 OK com [] ou lista de contas.

3.4 Endpoint: Buscar Conta por ID

  • Método: GET
  • URL: /accounts/{id}
  • Respostas:
    • 200 OK com objeto Account.
    • 404 Not Found com { "error": "Conta não encontrada" }.

3.5 Endpoint: Atualizar Conta

  • Método: PUT
  • URL: /accounts/{id}

Body (JSON):

{
  "name": "Novo Nome",
  "email": "novoemail@email.com"
}
  • Atualizar apenas campos enviados.
  • Não alterar cpf nem balance.
  • 200 OK com conta atualizada.
  • 404 Not Found se conta inexistente.

3.6 Endpoint: Deletar Conta

  • Método: DELETE
  • URL: /accounts/{id}
  • 200 OK com { "message": "Conta removida com sucesso" }.
  • 404 Not Found se conta inexistente.

3.7 Endpoint: Depósito

  • Método: POST
  • URL: /accounts/{id}/deposit

Body (JSON):

{
  "amount": 200
}
  • amount obrigatório e > 0.
  • Saldo final = saldo atual + amount.
  • 200 OK com { "message": "Depósito realizado", "balance": <novoSaldo> }.
  • 400 Bad Request se amount <= 0.
  • 404 Not Found se conta não existir.

3.8 Endpoint: Transferência

  • Método: POST
  • URL: /accounts/{id}/transfer

Body (JSON):

{
  "targetId": "uuid-destino",
  "amount": 100
}
  • targetId obrigatório.
  • amount obrigatório e > 0.
  • Origem e destino devem existir.
  • Origem ≠ destino.
  • Origem deve ter saldo suficiente.
  • 200 OK com { "message": "Transferência realizada com sucesso", "sourceBalance": ..., "targetBalance": ... }.
  • 400 Bad Request para saldo insuficiente, valor inválido ou contas iguais.
  • 404 Not Found se origem ou destino não existirem.

4. Requisitos Funcionais dos Frontends

4.1 Frontend Web (Desktop)

Página: frontend-web/index.html

Componentes principais:

  • Card “Criar Conta”:
    • Campos: Nome, E-mail, CPF, Saldo Inicial.
    • Botão “Criar Conta” (POST /accounts).
  • Card “Atualizar Conta”:
    • Campos: ID, Novo Nome, Novo E-mail.
    • Botão “Atualizar” (PUT /accounts/{id}).
  • Card “Receber Dinheiro (Depósito)”:
    • Campos: ID, Valor.
    • Botão “Depositar” (POST /accounts/{id}/deposit).
  • Card “Enviar Dinheiro (Transferência)”:
    • Campos: ID origem, ID destino, Valor.
    • Botão “Transferir” (POST /accounts/{id}/transfer).
  • Card “Buscar / Deletar Conta”:
    • Campo: ID.
    • Botão “Buscar” (GET /accounts/{id}).
    • Botão “Deletar Conta” (DELETE /accounts/{id}).
  • Card “Resposta da API”:
    • Área de texto exibindo JSON de resposta.
    • Estilo diferenciado para erro (fundo vermelho).
  • Seção “Todas as Contas”:
    • Botão “Atualizar Lista” (GET /accounts).
    • Tabela com ID abreviado, Nome, E-mail, CPF, Saldo.

4.2 Frontend Mobile-like

Página: frontend-mobile/index.html

  • Barra superior com título do app.
  • Navegação por abas:
    • 🏠 Início
    • ➕ Conta
    • 🔄 Transações
    • 🔍 Buscar

Aba Início:

  • Card “Saldo Total em Sistema” (soma de todos os balance via GET /accounts).
  • Lista de contas com nome, e-mail, CPF, saldo e ID.

Aba Conta:

  • Formulário com Nome, E-mail, CPF, Saldo Inicial.
  • Botão “Criar Conta” (POST /accounts).
  • Bloco de resposta com JSON.

Aba Transações:

  • Seção Depósito:
    • Campos: ID da Conta, Valor.
    • Botão “💰 Depositar” (POST /accounts/{id}/deposit).
  • Seção Transferência:
    • Campos: ID Conta Origem, ID Conta Destino, Valor.
    • Botão “🔄 Transferir” (POST /accounts/{id}/transfer).
  • Bloco de resposta compartilhado.

Aba Buscar:

  • Campo de ID.
  • Botão “Buscar” (GET /accounts/{id}).
  • Botão “Deletar Conta” (DELETE /accounts/{id}).
  • Bloco de resposta com JSON.

5. Requisitos Não Funcionais

  • Performance: adequada para uso local; sem SLAs rígidos.
  • Disponibilidade: sem alta disponibilidade (uso didático).
  • Persistência: dados em memória; reiniciar o servidor limpa as contas.
  • Segurança: sem autenticação/autorização.
  • Manutenibilidade: código simples e modular.

6. Fluxos Principais (E2E)

6.1 Fluxo: Criar Conta, Depositar e Transferir

  1. Criar Conta A com saldo inicial 1000.
  2. Criar Conta B com saldo inicial 200.
  3. Depositar 500 na Conta A.
  4. Transferir 300 da Conta A para Conta B.
  5. Verificar saldos finais:
    • Conta A: 1200.
    • Conta B: 500.

Esse fluxo é usado em testes de regressão, na collection do Postman e em demonstrações de ciclo de vida.

7. Critérios de Aceite

  • Todos os endpoints funcionando conforme especificação.
  • Frontend Web executa todas as operações básicas usando a API.
  • Frontend Mobile-like reflete corretamente os saldos (especialmente saldo total).
  • Swagger documenta todos os endpoints com exemplos.
  • Collection do Postman executa:
    • CRUD básico.
    • Depósito e transferência.
    • Fluxo E2E completo.
