Requisitos Funcionais — Banking API e Frontends

1. Contexto

O Banking API é um sistema didático, usado para demonstrar o ciclo de vida de desenvolvimento de software em um contexto simples de banco digital. Ele será utilizado em mentorias e treinamentos de QA, cobrindo desde a definição de requisitos até a execução de casos de teste e regressão.

2. Objetivos

  • Disponibilizar uma API REST estável e previsível para:
    • Criação, consulta, atualização e exclusão de contas.
    • Depósito de valores.
    • Transferência de valores entre contas.
  • Fornecer duas UIs simples:
    • Frontend Web Desktop: foco em visualização de dados e operações em cards.
    • Frontend Mobile-like: foco em navegação por abas e experiência de “app de banco”.

3. Escopo Funcional

3.1 Backend (API REST)

Entidade principal: Conta Bancária

Campos mínimos:

  • id (string, UUID gerado pelo sistema)
  • name (string, obrigatório)
  • email (string, obrigatório)
  • cpf (string, obrigatório, único)
  • balance (number, saldo atual)
  • createdAt (string, ISO datetime)

Requisitos gerais da API:

1. Criar conta

  • Método: POST /accounts
  • Body:
    • name: string (obrigatório)
    • email: string (obrigatório)
    • cpf: string (obrigatório, único)
    • initialBalance: number (obrigatório, >= 0)
  • Regras:
    • Não permitir initialBalance negativo.
    • Não permitir cpf duplicado.
  • Respostas:
    • 201 com conta criada (campo balance = initialBalance).
    • 400 em caso de campos ausentes, initialBalance inválido ou CPF duplicado.

2. Listar contas

  • Método: GET /accounts
  • Regras:
    • Retornar array de contas (possivelmente vazio).
  • Resposta: 200 com [] ou lista de contas.

3. Buscar conta por ID

  • Método: GET /accounts/{id}
  • Regras:
    • Se não encontrar, retornar 404.
  • Respostas:
    • 200 com a conta encontrada.
    • 404 com { "error": "Conta não encontrada" }.

4. Atualizar dados da conta

  • Método: PUT /accounts/{id}
  • Body:
    • name: string (opcional)
    • email: string (opcional)
  • Regras:
    • Atualizar apenas campos enviados.
    • Não permitir alterar cpf nem balance.
  • Respostas:
    • 200 com dados atualizados.
    • 404 se conta não existir.

5. Deletar conta

  • Método: DELETE /accounts/{id}
  • Regras:
    • Remover conta da memória se existir.
  • Respostas:
    • 200 com { "message": "Conta removida com sucesso" }.
    • 404 se conta não encontrada.

6. Depósito (receber dinheiro)

  • Método: POST /accounts/{id}/deposit
  • Body:
    • amount: number (obrigatório, > 0)
  • Regras:
    • Somar o valor ao balance da conta.
  • Respostas:
    • 200 com { "message": "Depósito realizado", "balance": <novoSaldo> }.
    • 400 se amount <= 0.
    • 404 se conta não existir.

7. Transferência (enviar dinheiro)

  • Método: POST /accounts/{id}/transfer
  • Body:
    • targetId: string (obrigatório)
    • amount: number (obrigatório, > 0)
  • Regras:
    • Validar existência de conta origem e destino.
    • Validar saldo suficiente na origem.
    • Proibir transferência para a mesma conta (id == targetId).
  • Respostas:
    • 200 com { "message": "Transferência realizada com sucesso", "sourceBalance": ..., "targetBalance": ... }.
    • 400 para saldo insuficiente, valor inválido ou origem=destino.
    • 404 se origem ou destino não existirem.

3.2 Documentação da API (Swagger)

  • Disponibilizar Swagger UI em /api-docs.
  • A documentação deve:
    • Descrever modelos de request/response.
    • Permitir executar chamadas diretamente.
    • Servir como base para teste manual de API.

3.3 Frontend Web (Desktop)

  • Exibir cards para:
    • Criar conta.
    • Atualizar conta.
    • Depósito.
    • Transferência.
    • Buscar/deletar conta.
  • Exibir “Resposta da API” em área própria (sucesso/erro).
  • Exibir tabela com ID abreviado, Nome, E-mail, CPF, Saldo.
  • Atualizar tabela após operações que alteram contas.

3.4 Frontend Mobile-like

  • Layout com navegação por abas:
    • 🏠 Início
    • ➕ Conta
    • 🔄 Transações
    • 🔍 Buscar
  • Aba Início:
    • Card de “Saldo Total em Sistema”.
    • Lista de contas com dados principais.
  • Aba Conta:
    • Formulário de criação de conta.
  • Aba Transações:
    • Formulário de Depósito e de Transferência.
  • Aba Buscar:
    • Buscar conta por ID.
    • Deletar conta por ID.

4. Restrições e Decisões

  • Armazenamento em memória (array em db.js).
  • Stack:
    • Backend: Node.js + Express.
    • Doc: Swagger.
    • Frontend: HTML + CSS + JS simples.
  • Sem internacionalização/multilíngue.

5. Critérios de Aceite (High Level)

  • Todos os endpoints descritos implementados e acessíveis.
  • Swagger acessível em /api-docs.
  • Ambos os frontends funcionando contra a mesma API.
  • Fluxos principais (criar conta, depósito, transferência, deletar) funcionando ponta a ponta.
